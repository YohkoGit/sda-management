## Successes

1. **Consistent public API pattern across all 5 stories** — Every endpoint followed `[AllowAnonymous]` + `[EnableRateLimiting("public")]` + explicit DTO whitelist + 200 with array for lists. By Story 5.2, the pattern was established and every subsequent story was predictable.
2. **Shared utility extraction at the right moment** — Story 5.3 extracted `formatActivityDate`, `formatTime`, `getDateLocale` from HeroSection into `lib/dateFormatting.ts` with 26 test cases. Also extracted `ExtractPredicateur` backend helper. Both reused across all subsequent stories with zero duplication.
3. **YouTube graceful degradation pattern** — `YouTubeService` NEVER throws to caller. All failure modes (quota exceeded, timeout, API error, missing key) degrade to `{ isLive: false }` + server-side logging. Established the gold standard for external API integration.
4. **Lazy loading for bundle splitting** — Story 5.5 converted PublicCalendarPage to `lazy()` import with Suspense, preventing 80-150KB of Schedule-X + temporal-polyfill from loading on every page.
5. **DTO reuse** — `PublicActivityListItem` record used for both upcoming activities (5.3) and calendar events (5.5). Clean pattern avoiding DTO proliferation.
6. **Accessibility blind spot improved** — ARIA/accessibility issues dropped from 3/8 stories in Epic 4 to 1/5 in Epic 5 (only 5.1 had a redundant `aria-label` caught in review). The assertion checklist from Epic 4 is working.
7. **DatabaseSeeder infrastructure fix** — Pre-existing `SeedDevDataAsync()` issue (called during host startup before test migrations) surfaced in 5.2 and was fixed with `SeedDevData` config guard. Benefits all future integration tests.
8. **Well-reasoned architecture deviation** — Story 5.2 deviated from original architecture doc (simple URL config) to use YouTube Data API v3 for live detection, backed by ADR analysis (First Principles, Red Team/Blue Team, What-If Scenarios). Improved product trustworthiness.

## Challenges

1. **Error/retry state handling forgotten during implementation (2/5 stories)** — Story 5.3: ProgramTimesSection had no error handling. Story 5.5: calendar had no retry button AND no department-fetch error state. Both caught in code review, not implementation.
2. **Additional test cases discovered during code review (3/5 stories)** — Story 5.1: date formatting tests needed. Story 5.3: error state test needed. Story 5.5: start>end boundary test needed. Tests are being added reactively in review rather than proactively during implementation.
3. **Third-party library version/type surprises (1/5 stories)** — Story 5.5: Schedule-X v4.3.1 requires `temporal-polyfill@0.3.0` exactly (caret range pulls incompatible 0.3.2). `CalendarEventExternal` requires `ZonedDateTime|PlainDate`, not `PlainDateTime`. Sonner Toaster renders `<section role="region">` conflicting with test queries.
4. **Pre-existing DatabaseSeeder interference (1/5 stories)** — Story 5.2: `SeedDevDataAsync()` called during host startup before test migrations, causing integration test failures. Hidden infrastructure assumption from earlier epics.
5. **Dead code / redundant imports (2/5 stories)** — Story 5.1: skeleton test dead conditional. Story 5.3: redundant Tailwind classes and EF Core calls. Minor but recurring.

## Key Insights

1. **Consistent patterns accelerate velocity** — The public endpoint → DTO → service → hook → component pattern made every story predictable. Established by Story 5.2, it carried the entire epic. Pattern consistency is a force multiplier.
2. **Error state handling is the new blind spot** — Same evolution as accessibility was in Epic 4. Accessibility improved with the checklist; error states need the same treatment. The fix is expanding the assertion checklist, not adding a new process.
3. **Shared utility extraction timing matters** — Extracting too early creates premature abstractions. Extracting too late means duplication. Story 5.3 hit the sweet spot: two stories had used the same logic, so extraction was justified and immediately reused.
4. **External API integration requires defensive design** — The YouTube pattern (never throw, degrade to safe default, log server-side) should be the standard for all future external integrations.
5. **Third-party integrations carry hidden version costs** — Radix in Epic 3, React Hook Form in Epic 4, Schedule-X in Epic 5. Each epic's third-party library had its own gotchas. Up-front version compatibility checks and type definition review can reduce surprises.
6. **Code review remains the essential safety net** — Every issue in Epic 5 was caught before shipping. The question is shifting left: can we catch error states, boundary tests, and dead code during implementation rather than review?
