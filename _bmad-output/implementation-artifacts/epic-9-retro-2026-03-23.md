## Successes

1. **Push-only hub with zero client-callable methods** — Clean separation: mutations via REST API, notifications via SignalR. The hub only manages connection lifecycle (OnConnectedAsync group assignment). All broadcasting happens via IHubContext<NotificationHub> in services. No bidirectional complexity.
2. **Lightweight signal DTOs prevent data leakage** — Intentional deviation from architecture's "payload is full read DTO" guidance. The public group never receives authenticated field values (ZoomLinks, role assignments, staffing details) through SignalR. Clients re-fetch via their own auth-scoped REST endpoints after receiving the invalidation signal. Security by design.
3. **Clean state boundary: Zustand (UI) vs TanStack Query (server)** — Badge store tracks only `activityId → ISO timestamp` pairs. Activity data remains exclusively in TanStack Query. No state store contamination. Zustand persist middleware handles localStorage with partialize for selective persistence.
4. **Error isolation at every layer** — Backend: broadcast failure logged as warning, doesn't fail REST response. Frontend: markModified wrapped in try-catch, doesn't block cache invalidation. This mirrors the YouTube graceful degradation pattern from Epic 5.
5. **Context.User over ICurrentUserContext in hub** — Correct understanding that SignalR hubs are transient with per-invocation DI scope. CurrentUserContextMiddleware populates ICurrentUserContext in the HTTP request scope, but the hub gets a fresh empty instance. Using Context.User (ClaimsPrincipal) + AppDbContext query mirrors the middleware's logic correctly.
6. **stopConnection leak fix** — Code review in 9.1 revealed original stopConnection only stopped Connected state, leaking Reconnecting/Connecting connections. Fixed to stop any non-Disconnected state with reference capture before nulling to prevent race conditions.
7. **"Trust transfer" framing elevates badge to core feature** — UX spec positions "Modifié" as replacing social proof in a church context. Not polish — it prevents someone giving wrong information based on stale data. amber-500 (attention/change) is semantically correct per UX spec status colors.

## Challenges

1. **ICurrentUserContext unavailable in SignalR hub** (9.1) — The #1 gotcha of this epic. SignalR hubs are transient with per-invocation DI scope. CurrentUserContextMiddleware populates ICurrentUserContext in the HTTP request scope, but the hub gets a fresh, unpopulated instance (IsAuthenticated=false, Role=Anonymous). Required using Context.User (ClaimsPrincipal) + AppDbContext query instead.
2. **Visibility string casing mismatch** (9.2) — ActivityResponse.Visibility uses lowercase ("public"/"authenticated") but enum .ToString() produces PascalCase ("Public"/"Authenticated"). Required StringComparison.OrdinalIgnoreCase in ResolveGroup. Codebase inconsistency between serialized DTO values and enum string representations.
3. **Date-sensitive test failures** (9.1) — DayDetailDialog tests from Epic 7 used March 2026 dates that became past on March 22, 2026. Fixed by shifting to 2027. This is a project-wide anti-pattern — hardcoded future dates eventually become past dates.
4. **React Strict Mode complexity for event systems** (9.2) — Double-mount causes callback accumulation on onreconnected. No public API to remove onreconnected callbacks. Required WeakSet guard to track registered connections. Also requires conn.off(event) before conn.on(event) to prevent duplicate handlers.
5. **NSubstitute Arg.Is<> limitation** (9.2) — Doesn't support C# `is` pattern-matching in expression trees. Refactored unit tests to captured message list with post-hoc assertions.
6. **Sonner `<section>` test interference** (9.3) — container.firstChild picks up Toaster's section element. Consistent with previous epics (5.5, 7.1). Fixed with specific screen.queryByText assertions.

## Key Insights

1. **SignalR DI scoping is the most important non-obvious constraint** — The difference between "hub DI scope" (transient, per-invocation) and "request DI scope" (middleware-populated) is the #1 source of bugs in real-time features. This must be documented prominently in codebase conventions.
2. **Lightweight signal DTOs > full entity broadcasting** — Three benefits: security (no data leakage across group boundaries), performance (smaller payloads), simplicity (client always re-fetches via REST using its own auth context). The architecture's "payload is full read DTO" guidance was correctly overridden with explicit rationale.
3. **Date-sensitive test data is a project-wide anti-pattern** — Fixed reactively in 9.1, but the root cause is hardcoded dates like "2026-03-21" in test fixtures. The FutureDate() helper from Story 8.4 should be used universally. All test dates should be relative to `Date.now()`.
4. **React Strict Mode creates real engineering complexity for event systems** — WeakSet guard, conn.off before conn.on, void operator for floating promises, poll interval for connection availability — all patterns needed to handle double-mount correctly. This complexity is inherent to the React + SignalR integration, not a code quality issue.
5. **Error isolation is a pattern, not a one-time decision** — Backend broadcast failure (9.2) mirrors frontend badge failure (9.3) mirrors YouTube API failure (5.2). The principle "secondary operation failure must not fail primary operation" applies consistently across the entire stack.
