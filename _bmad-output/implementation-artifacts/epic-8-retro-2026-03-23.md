## Successes

1. **Meetings as Activity records — elegant domain modeling** — No separate Meeting entity, no MeetingsController, no separate pipeline. 5 nullable fields on the existing Activity entity keep the pipeline unified. Same queries, same cache keys, same StaffingIndicator (hidden for meetings). The IMeetingRequest interface + MeetingValidationRules kept validation DRY without polluting the shared IActivityRequest.
2. **Comprehensive OWNER verification (8.4)** — 20 integration tests + 4 frontend tests proving OWNER bypass on unassigned departments. Guard assertion (`OwnerHasZeroDepartmentAssignments`) prevents future regressions. Anti-pattern audit across 14 frontend files confirmed all `departmentIds` usages include correct OWNER bypass. Zero production code changes needed — the system was already correct.
3. **BadRequestException infrastructure (8.3)** — Reusable exception class + GlobalExceptionHandler maps business rule violations to 400 ProblemDetails with `urn:sdac:validation-error` type. Clean separation: FluentValidation → 422 for structural validation, BadRequestException → 400 for domain business rules (e.g., lead not in department).
4. **Two-query aggregate staffing (8.1)** — CriticalGap detection requires role-level data per activity (can't be in SQL GROUP BY). Pragmatic approach: query departments + query activities with roles, compute in-memory via existing `ComputeStaffingStatus()`. Efficient for ~12 departments × ~50 activities.
5. **Component reuse pattern: SubMinistryManager** — Enhanced once with lead picker (shadcn Select + officer data fetching), reused in both AdminDepartmentsPage (OWNER context) and DepartmentDetailPage (ADMIN+ context). Self-contained data fetching via internal useQuery.
6. **placeholderData from list cache (8.1)** — TanStack Query `placeholderData` from list cache provides instant department detail header while the full query (with sub-ministries) loads. `isPlaceholderData` flag enables subtle loading indicator for sections not in the cache.
7. **Authorization widening done safely (8.1, 8.3)** — Activities GET filtered branch relaxed from CanManage to CanView (VIEWERs can already access individual activities). Sub-ministry CRUD changed from IsOwner to CanManage (ADMIN+ with department scope). Clear security analysis in story notes.

## Challenges

1. **Build lock issue blocking EF Core migrations** (8.1, 8.2) — API process (running dev server) held file locks preventing `dotnet ef migrations add`. Had to kill the process and retry. Recurring friction point for stories with migrations.
2. **AxiosResponse unwrap pattern missed again** (8.3) — SubMinistryManager's queryFn didn't unwrap `.data` from AxiosResponse. Lead picker rendered empty. Caught in UI validation as HIGH bug. This is the same pattern issue from Epic 6 (Story 6.1). Despite being documented as a team agreement, it was missed in a new component.
3. **Story 8.3 had 4 HIGH code review issues** — Lead validation errors not surfaced in UI (H1), missing mutation payload tests (H3), UpdateAsync didn't populate lead fields in response (H4). The sub-ministry + lead combination had the highest surface area of any story in the project.
4. **JSON `WhenWritingNull` serialization** (8.2) — Backend omits null fields from JSON responses. Integration tests needed `TryGetProperty` instead of checking `ValueKind == Null`. Codebase-specific gotcha documented in previous retros but still causes friction.
5. **IMeetingRequest interface separation** (8.2) — Meeting fields must NOT go on the shared IActivityRequest interface. Required careful DTO design to keep the shared ActivityValidationRules meeting-agnostic while adding meeting-specific validation to concrete validators only.
6. **Test setup criticality (8.4)** — The OWNER test user must have zero department assignments for tests to prove bypass (not just scope matching). Guard assertion prevents silent regression. The framing "test setup matters more than tests" is a valuable principle.

## Key Insights

1. **Extending existing entities > creating new ones** — Meetings as Activity (5 fields), Lead as SubMinistry FK (1 field). Keeps pipelines unified, avoids entity proliferation, reuses existing controllers/services. Only extend when the new concept is genuinely a variant of the existing one (meetings ARE activities with extra context).
2. **Verification stories reveal test infrastructure quality** — Story 8.4 wrote 24 tests and changed zero production code. The value was: (a) proving the system works, (b) guard assertions preventing regression, (c) anti-pattern audit confirming frontend safety. This pattern should be used for other cross-cutting concerns.
3. **The AxiosResponse unwrap pattern is the project's most persistent gotcha** — Epic 6 normalized it, Epic 8.3 missed it. It needs to be in the implementation checklist, not just the team agreements. Every new `queryFn` and service method must be verified.
4. **Business rule exceptions need a dedicated path** — FluentValidation (structural, 422) vs BadRequestException (domain rules, 400) is a clean separation. The GlobalExceptionHandler pattern is reusable for future business rule validations.
5. **Story complexity correlates with entity modification breadth** — Story 8.3 touched entity, DTO, validator, service, controller, frontend component, page, schema, tests. 11 code review issues. Story 8.4 (verification only) had 7 issues. The more layers you touch, the more gaps appear.
