## Successes

1. **3 well-documented ADRs** — Year view strategy (simplified YearGrid over over-engineered 300-400 line custom calendar), backend filtering over frontend `setCalendars()`, unified Day Detail Dialog for all roles. Each had clear rationale, trade-off analysis, and rejected alternatives documented.
2. **Component extraction with zero regressions** — CalendarView extracted from PublicCalendarPage (7.1) and ActivityForm extracted from AdminActivitiesPage (7.3). Both extractions verified by running full existing test suites immediately after extraction. Clean pattern.
3. **Dual-renderer architecture for Year view** — Schedule-X v4 has no Year view. Rather than fighting the library, built a custom YearGrid (~160 lines) that renders alongside Schedule-X. Single `selectedDate` source of truth prevents state sync bugs. Creative and pragmatic.
4. **Backend filtering over frontend filtering** — Department filter via `departmentIds` query parameter reduces payload, works identically for both Schedule-X views and custom YearGrid, and simplifies frontend. The ADR analysis showed frontend-only filtering would break month grid "+N more" indicators.
5. **Unified Day Detail Dialog** — Same gesture for all roles (click day → see day detail). ADMINs see "Nouvelle activité" button for step 2 (template) and step 3 (form). Better UX than role-divergent click behavior. Focus group analysis showed ADMINs prefer seeing day context before creating.
6. **Prop-driven CalendarView control** — `navigateTo` prop for programmatic view/date changes fits unidirectional data flow. Rejected `useImperativeHandle` as overkill. `onDayAction` callback is optional and backward-compatible.
7. **Calendar-controls plugin verification spike** — Story 7.1 started with a Task 0 verification spike to confirm `@schedule-x/calendar-controls` API availability before building dependent features. Risk mitigated early.

## Challenges

1. **Schedule-X v4 has significant gaps** — No Year view (`createViewYear()` doesn't exist), no built-in `setView()`/`setDate()` on CalendarApp instance. Required installing separate `@schedule-x/calendar-controls` plugin and building a custom YearGrid component. More integration work than the architecture doc implied.
2. **Highest code review issue count across all epics: 25** — 7.1: 3 HIGH (year data flow not wired to hooks, loading/error states missing for year view). 7.2: 1 HIGH (missing auth endpoint tests), 4 MEDIUM (DepartmentFilter placement, useCallback deps, roving tabindex, null guard). 7.3: 2 CRITICAL (missing tests for 3 tasks, department error handling in form), 3 MEDIUM (generic error toast, full creation flow test, mock pollution).
3. **Data flow wiring is the persistent blind spot** — Story 7.1's 3 HIGH issues were all about connecting data hooks to UI states. `onYearChange` wasn't wired through CalendarView to page wrappers. Year view loading/error states weren't surfaced. The visual layer was correct but the data pipes were disconnected.
4. **Predicateur extraction duplicated between services** — `CalendarService` duplicates `PublicService`'s private `ExtractPredicateur()` method. Both do case-insensitive "Predicateur"/"Prédicateur" role search. Accepted for 2 call sites — extract to shared helper if a 3rd appears.
5. **Year view data fetching requires 4 parallel API calls** — Backend caps `GET /api/public/calendar` at 90 days. Year view needs 365 days. Solved with 4 quarterly `useQuery` calls merged via the hook. Works but adds complexity and potential for partial failures.

## Key Insights

1. **ADR-driven decisions produce better outcomes** — Each ADR was well-reasoned and none were regretted during implementation. The Year view ADR saved significant development time (simplified ~160 lines vs estimated 300-400 lines). The backend filtering ADR prevented a subtle bug (frontend-only filtering would have broken month grid overflow indicators). ADRs should continue for non-obvious architectural choices.
2. **Code review issue count correlates with architectural complexity, not quality decline** — Epic 7 was the most architecturally complex epic (dual renderers, 3-step dialog with responsive surface switching, component extractions, new backend controller). The 25 issues reflect the inherent complexity of calendar integration, not a regression in implementation quality.
3. **HIGH/CRITICAL review issues cluster around data flow wiring** — Across all 3 stories, the visual/UI layer was consistently correct. The gaps were in connecting data hooks to loading/error/success states and wiring callbacks through component hierarchies. This is the same pattern from Epic 6 (error states) but at a deeper level (data pipe connections).
4. **Component extraction is safe when existing tests verify behavior** — Both CalendarView and ActivityForm extractions were verified by running the full existing test suite immediately after extraction. The key insight: extraction is a refactoring operation, not a feature addition. Existing tests are the safety net.
5. **Schedule-X v4 is useful but not complete** — The library handles Day/Week/Month views well but has gaps (no Year view, no programmatic control without plugin, locale set at init time). For calendar-heavy features, expect to supplement with custom components. This isn't a problem — it's a pattern: use the library for what it does well, build custom for what it doesn't.
