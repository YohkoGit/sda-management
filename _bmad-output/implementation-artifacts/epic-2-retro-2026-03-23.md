## Successes

1. **Consistent CRUD architecture across all 6 stories** — Every admin page follows the same pattern: empty state → card grid → form dialog → toast confirmation. This consistency compounds across epics for maintainability and onboarding.
2. **Sanitization pipeline reorder (Story 2.1)** — Moving sanitization before validation (sanitize → validate → save) caught a real security hole where HTML-only inputs could bypass `NotEmpty()` checks. Became the standard for all subsequent stories.
3. **Dev experience improvements (Story 2.2)** — Rate limit bump from 5 → 100 req/min, `dev.sh` script, and role case normalization (`.toUpperCase()`) eliminated friction for all subsequent development.
4. **Previous Story Intelligence sections** — Every story's dev notes included PSI sections that prevented rediscovery of patterns and accelerated implementation.
5. **Test infrastructure continued to compound** — `IntegrationTestBase` from Epic 1 enabled every Epic 2 story to have real PostgreSQL tests instantly. Grew from ~200 to 404 total tests.
6. **Tooling version prerequisites eliminated surprises** — All 6 stories had explicit prerequisites tables. Zero tooling version surprises (vs. Epic 1 which had Vite 7, .slnx, FluentValidation v12, Zod v4 issues).

---

## Challenges

1. **Code review findings not feeding forward** — Same issues repeated across stories because learnings stayed in the review and didn't flow into the next story's dev notes. The getById-from-card pattern appeared in Stories 2.3 AND 2.4 — same mistake, same fix, back to back.
2. **Missing authorization tests repeated in 4/6 stories** — Viewer 403 and anonymous 401 tests were consistently missed during initial implementation and caught in code review (Stories 2.1, 2.3, 2.4, 2.5).
3. **Two unfulfilled action items from Epic 1** — Test assertion checklist (never created) and manual QA checklist (never created). The missing test checklist directly caused the most frequent review finding category.
4. **Cross-model review agreement unclear** — Epic 1 retro established "different AI model for review vs implementation" as default, but Epic 2 stories don't document which model performed reviews.

---

## Key Insights

1. **The feedback loop between code review and next story is broken** — Learnings stay in the review, not in the dev notes. This is the root cause of repeated findings.
2. **Unfulfilled retro action items have real consequences** — The missing test assertion checklist directly caused 4/6 stories to have the same review finding.
3. **Cross-model review must be strictly enforced and documented** — No ambiguity about which model reviewed. Team agreement upgraded to strict enforcement.
4. **Manual QA debt is growing** — 9 epics, 400+ automated tests, zero manual verification. This is the single production deployment blocker.
