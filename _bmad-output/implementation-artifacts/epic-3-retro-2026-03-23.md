## Successes

1. **Code review feedback loop fully operational** — Every story from 3.2 onward had "Previous Story Learnings" sections that explicitly warned about `db.Set<UserDepartment>()`, `AuthUser.userId` vs `UserResponse.id`, and Radix jsdom polyfills. Prevented repeated pitfall rediscovery. This was the biggest process improvement carried from Epic 2.
2. **Full user lifecycle in 5 stories** — Create (single + bulk), edit with role/department management, soft-delete, avatar upload. Complete admin experience delivered at 100% completion.
3. **DepartmentMultiSelect extraction** — Shared component eliminated ~50 lines of duplication across UserFormDialog and BulkUserFormDialog. Compounding reuse benefit.
4. **Auth test coverage improved** — Missing auth tests dropped from 4/6 stories (Epic 2) to 2/5 stories (Epic 3). PSI sections helped carry lessons forward.
5. **Breakthrough patterns established:**
   - Global query filter for soft-delete (one line protects all auth flows)
   - Four-level auth (authenticate → role → self-guard → scope)
   - Variable-based mutationFn (solves Radix AlertDialog stale closure)
   - Post-materialization DTO pattern (file system calls outside EF Core .Select())

---

## Challenges

1. **Radix/jsdom polyfills rediscovered in 3/5 stories** — `hasPointerCapture`, `setPointerCapture`, `releasePointerCapture`, `scrollIntoView` needed in Stories 3.3, 3.4, and 3.5. No shared test utility created — each story copy-pasted the setup.
2. **Guest user filtering missed in 2/5 stories** — Story 3.1 (GetByIdAsync) and Story 3.5 (avatar upload) both needed guest filtering added during code review. Easy to miss pattern.
3. **Error handling gaps caught as HIGH severity** — Null IFormFile (3.5 H1), ImageFormatException for spoofed content types (3.5 H2), non-Axios error fallback (3.4 M2). All caught in review, not during implementation.
4. **Commit hygiene: Story 3.4 + 3.5 changes mixed** — Code review for 3.5 flagged that Story 3.4 changes were still uncommitted and interleaved. Makes review harder and risks losing track of changes.
5. **MSW + FormData + jsdom incompatibility** — Avatar upload tests in Story 3.5 couldn't use MSW (mutations hung). Had to mock at service level, reducing test realism.

---

## Key Insights

1. **The feedback loop works — formalize it further** — PSI sections are effective but informal. A formal assertion checklist would prevent the guest filtering and error handling gaps that still slipped through.
2. **Radix testing workarounds must be systematized** — Copy-pasting polyfills per test file is a process failure. Extract into shared test utility.
3. **Global query filter is an elegant pattern** — Single `HasQueryFilter(u => u.DeletedAt == null)` automatically protects every auth flow. Worth celebrating and replicating if other entities need soft-delete.
4. **Manual QA deferred intentionally** — Strategy decided: complete all epic retrospectives first (Epics 1–9), then comprehensive manual QA before deployment.

## Debug Highlights

| Story | Issue | Resolution |
|---|---|---|
| 3.1 | `db.Set<UserDepartment>()` — no DbSet property | Used `db.Set<UserDepartment>()` instead |
| 3.1 | Email max length boundary (255 vs 256) in Zod | Adjusted test string length |

| 3.3 | TS type casting for `editUser.role` in form reset | Cast to `CreateUserFormData["role"]` |
| 3.4 | Radix AlertDialog stale closure bug | Variable-based `mutationFn` with `mutate(user!.id)` |
| 3.4 | "Found multiple elements" in test | More specific regex: `/irréversible.*Marie-Claire Legault/` |
| 3.5 | ImageSharp `UnknownImageFormatException` on random bytes | Generate valid JPEG images in test helper |
| 3.5 | Oversized file test (600KB) exceeded MultipartBodyLengthLimit | Used 513KB (between MaxFileSizeBytes and limit) |
| 3.5 | MSW + FormData + jsdom mutations hang | Mock at service level with `vi.spyOn` |
| 3.5 | Explicit Content-Type header without boundary | Removed — axios auto-detects FormData |
